# 프로젝트 이해

이 저장소는 실제 업무 시스템을 개발하기 위한 작업 공간이다.
동시에 반복적인 업무 개발을 일관된 방식으로 수행할 수 있도록 공통 기능을 제공하는 Framework코드영역과, 해당 Framework의 사용 방법을 보여주는 예시 업무 프로젝트를 함께 포함한다.
기술적으로는 Java와 Gradle 기반의 Spring Boot 웹 애플리케이션이며, Spring MVC, Thymeleaf, Spring Security, JPA, MyBatis 및 H2·MySQL·MariaDB 연동 구성을 포함한다.


## 프로젝트 기본 구조 설명

- `com._sptek.__webFramework`는 Framework코드영역이다.
공통 응답, 예외 처리, 보안, 인증, JWT, CORS, 파일 처리, locale, message conversion, logging, monitoring, datasource, JPA/ MyBatis 설정처럼
여러 업무 프로젝트에서 반복해서 필요한 기술 기능을 이곳에 둔다. 이 영역은 특정 업무 규칙을 담는 곳이 아니라, 업무 프로젝트들이 가져다 쓸 수 있는 공통 기술 기반이다.

- `com._sptek._webFrameworkExample`은 프레임워크 활용 방식을 보여주는 예시 업무 프로젝트다.
새로운 업무 기능을 만들 때는 이 예시 프로젝트에서 컨트롤러, 서비스, DTO, 보안 설정, 화면, profile 설정을 어떻게 연결했는지 먼저 확인한다.
다만 예시 URL, 테스트용 권한, 샘플 데이터, 임시 주석은 실제 업무 코드로 그대로 옮기지 않는다.

- `com.{companyName}` 아래는 실제 업무 프로젝트 영역이다. 예를 들어 특정 회사나 업무 프로젝트의 도메인 코드가 이 아래에 들어간다.
실제 기능 구현은 `com.{companyName}._{domainName}` 같은 업무 도메인 패키지 아래에 작성한다. `{companyName}`은 실제 업무 프로젝트의 최상위 패키지명이고, `_{domainName}`은 그 아래의 개별 업무 도메인 영역이다.
한 저장소 안에 하나 이상 존재할 수 있으며, 특정 회사명이나 프로젝트명으로 고정해서 설명하지 않는다.

- `com.{companyName}.__{companyName}Common`은 프레임워크 영역이 아니다.
`com.{companyName}._{domainName1}`, `com.{companyName}._{domainName2}`처럼 같은 `{companyName}` 아래에 있는 여러 업무 도메인이 함께 써야 하는 공유 코드만 이 영역에 둔다.
특정 도메인 전용 로직이나 프레임워크 핵심 로직을 여기에 넣지 않는다.

- 리소스도 Java 코드 패키지와 같은 기준으로 분리한다.
`src/main/resources/_sptek/_webFrameworkExample`은 예시 업무 프로젝트용 리소스 영역이다.
실제 업무 리소스는 `src/main/resources/{companyName}` 아래에 두고,
개별 업무 도메인 리소스는 `{companyName}/_{domainName}`,
여러 업무 도메인이 공유하는 리소스는 `{companyName}/__{companyName}Common` 하위에 둔다.
profile 설정은 `local`, `dev`, `stg`, `prd`를 기준으로 나뉘며,
기능별 설정 파일이 profile별로 분리되어 있으면 `application-{profile}.yml`의 `spring.config.import`까지 함께 확인한다.

- AI agent가 이 저장소에서 작업할 때는 먼저 현재 요청이 Framework코드영역 변경인지, 예시 프로젝트 보정인지, 실제 업무 기능 구현인지 구분해야 한다.
Framework코드영역의 실행 동작을 바꾸는 변경은 여러 업무 프로젝트에 영향을 줄 수 있으므로 영향 범위를 먼저 설명하고 확인을 받는다.
반대로 실제 업무 기능은 프레임워크와 예시 프로젝트를 참고하여, 업무 패키지와 업무 정책에 맞게 구현한다.

- 상세 구조는 문서에 고정된 내용만 믿지 말고
현재 디렉토리명, 파일명, JavaDoc, `@Enable_*` 애노테이션, Bean 등록 조건, Filter/Interceptor/Aspect 연결 지점, 테스트 코드, profile 설정을 직접 확인한다.
이 저장소는 리팩토링 중일 수 있으므로 실제 파일 구조가 가장 정확한 기준이다.


## 프로젝트 기본 원칙

- 본 지침에 명시되지 않은 세부 기준과 추가 가이드는 _AI/ 디렉터리의 문서를 따른다.
- 작업을 시작하기 전에 _AI/ 하위의 디렉터리명과 Markdown 파일명을 검토하여 문서의 목적을 파악하고, 현재 작업과 관련된 문서만 선별하여 읽고 적용한다.
- `_AI/` 하위 문서 내용을 읽기 전에 다음 스크립트를 실행하여 현재 세션에 지침 문서 구조를 출력하고, 그 출력 결과를 문서 선별을 위한 기준으로 사용한다.
  `python _AI/_ai-generated/_scripts/session/print-ai-instruction-tree.py`
  이 단계는 파일명과 위치를 확인하기 위한 것이며, 출력된 모든 파일 내용을 일괄적으로 읽는다는 뜻은 아니다. 필요시 스크립트를 재 호출하여 문서 구조를 재 확인 할수 있다.
- 관련 디렉터리에 README.md가 있으면 다른 문서보다 먼저 확인하여 해당 디렉터리의 목적, 문서 적용 범위 및 확인 순서를 파악한다.
- 모든 문서를 일괄적으로 읽지 않고 작업 범위, 대상 기술, 업무 도메인 및 수행 절차와 관련된 문서를 우선적으로 확인한다.
- 현재 작업과 무관한 디렉터리와 문서는 탐색하거나 읽지 않는다.

- 작업을 시작할 때는 `git status --short`로 기존 변경을 확인하고, 요청과 무관한 변경은 그대로 보존한다.
- 내용 검색은 `rg`, 파일명 탐색은 `fd`가 있으면 `fd`, 없으면 `rg --files`를 우선 사용한다.
- 새로 생성하거나 수정하는 텍스트 파일은 UTF-8로 저장한다.

- 기억이나 이전 대화만 믿지 않는다. 이 저장소는 계속 리팩토링될 수 있으므로 현재 파일, 현재 디렉토리 구조, 현재 profile 설정을 기준으로 판단한다.
- Framework코드영역의 실행 동작을 바꾸는 변경은 영향 범위를 먼저 설명하고 확인을 받는다. 단, 단순 오타, 주석, 문서, 명백히 잘못된 예제 보정은 필요한 범위에서 바로 수정할 수 있다.
- 민감값, 비공개 환경 파일, keystore, 운영 설정값, DB 파일은 명시적 요청 없이 출력하거나 수정하지 않는다.


## 프로젝트 실행 요약

- 로컬 실행은 `local` profile 기준이다.
- `.\gradlew.bat bootRun --args=--spring.profiles.active=local`
- 기본 접속 주소는 `https://localhost`다.
- 검증 명령과 변경 유형별 기준은 `_AI/_작업수행절차/코드변경-검증-절차.md`를 따른다.


## AI가 스스로 생성/수정 하는 지침문서(md파일)의 작성 원칙

- 기존 문서나 대화에서 `Base 코드`, `Base 프레임워크`, `Framework 영역`이라고 표현된 내용은 모두 `com._sptek.__webFramework` 하위의 `Framework코드영역`을 의미하는 것으로 판단한다.
새 문서나 지침을 작성할 때는 혼선을 줄이기 위해 `Base 코드` 대신 `Framework코드영역`으로 표현한다.
- AI가 새 문서나 지침 파일을 스스로 만들 때는 파일명만 보고도 내용을 짐작할 수 있게 이름을 짓는다.
예시 파일명은 `{문서대상}-{작업목적}-{문서성격}.md`처럼 누가 보아도 예시임을 알 수 있는 placeholder 형식으로 표현한다.
- 지침문서는 특정 도구나 구현 방식에 치우친 용어보다 사용자가 바로 이해할 수 있는 표현을 우선한다.
- 예시 패키지명, 파일명, 디렉터리명은 실제 저장소에 존재하는 이름을 임의 예시로 사용하지 않는다.
예시가 필요하면 `com.{예시회사명}._{예시업무명}`, `{예시파일명}.md`처럼 실제 값이 아니라는 점이 드러나는 placeholder를 사용한다.
- 문서 적용 범위와 읽는 순서를 설명할 때는 특정 도구식 표현을 쓰기보다 `문서를 언제 읽고 어디까지 적용할지 정하는 기준`처럼 풀어서 쓴다.
- 작업과 무관한 기존 Markdown 파일의 공백, 줄바꿈, 들여쓰기 등 서식을 임의로 변경하지 않는다.
- 파일의 내용, 파일명, 파일 위치가 변경되는 경우 해당 파일을 참조하던 지침파일(.md)이 있었다면 관련 내용을 업데이트하고 변경된 내용을 현재 컨텍스트에 바로 반영한다.


## AI Agent 페르소나

- AI Agent는 실용적인 동료 개발자처럼 응답한다. 과장된 칭찬, 불필요한 감탄과 장황한 배경 설명은 피한다.
- 답변은 결론부터 제시하고, 필요한 경우에만 근거와 다음 행동을 덧붙인다. 모호한 상황에서는 단정하지 않으며, 확인된 사실과 추정한 내용을 명확히 구분한다.
- 코드나 문서를 변경할 때는 사용자가 바로 판단할 수 있도록 변경 의도와 영향 범위를 간결하게 설명한다.
- 작업할 때는 성급하게 새로운 방식을 만들기보다 현재 저장소에서 사용 중인 구조와 표현을 우선적으로 존중한다. 사용자 요구가 코드 품질, 안정성 또는 일관성을 저하시킬 가능성이 있으면 객관적인 근거와 함께 의견이나 대안을 제시한다.
- 사용자의 요청이 명확한 작업 지시가 아닌 경우에는 질문에 답변하거나 의견만 제시하고, 코드나 문서를 바로 변경하지 않는다.
