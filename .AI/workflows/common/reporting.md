# Reporting Workflow

이 문서는 이 저장소의 커밋 메시지 제안과 완료 보고 절차를 정의한다.

## 완료 보고

- 먼저 구현 결과를 짧게 설명한다.
- 변경한 파일과 핵심 설계 결정을 밝힌다.
- 실행한 검증 명령과 결과를 정확히 기록한다.
- 실행하지 못한 검증은 이유와 함께 구분한다.
- 남은 위험, TODO, 사용자 결정이 필요한 사항이 있을 때만 후속 항목으로 제시한다.
- 프로젝트 근거와 일반적인 권고를 섞지 말고 어느 쪽인지 명확히 구분한다.

## 커밋 메시지

- 커밋 메시지는 한글로 작성한다. Agent가 직접 커밋할 때도 이 원칙을 따른다.
- 커밋 메시지 생성 요청을 받으면 현재 diff를 확인하고 변경 의도 중심의 한글 메시지를 제안한다.
- 제목은 간결한 한 문장으로 작성한다.
- 필요한 경우 본문에 주요 변경점과 검증 결과만 짧게 적는다.
- 단순 변경은 제목만 제안한다.
- 민감값, 개인 경로, 임시 파일, 생성 파일은 커밋 메시지에 포함하지 않는다.

### 커밋 메시지 작성 형식

- 제목은 "무엇을 바꿨는지"보다 "어떤 의도로 변경했는지"가 드러나도록 작성한다.
- 제목은 마침표 없이 한 문장으로 작성한다.
- 본문은 변경이 여러 파일이나 여러 책임에 걸쳐 있을 때만 작성한다.
- 본문에는 핵심 변경점, 유지한 동작, 영향 범위, 검증 결과만 포함한다.
- 검증을 실행하지 않았으면 검증한 것처럼 쓰지 않는다. 필요한 경우 `검증: 미실행` 또는 완료 보고에서 미실행 사유를 분리해 적는다.
- 커밋 메시지에는 파일 목록을 그대로 나열하기보다 변경 의도와 책임 단위를 요약한다.

단순 변경은 제목만 사용한다.

```text
async 예제의 비동기 처리 기준 주석을 보강한다
```

본문이 필요한 변경은 다음 형식을 사용한다.

```text
{변경 의도를 한 문장으로 작성한다}

- 주요 변경점 1
- 주요 변경점 2
- 영향 범위 또는 유지한 동작

검증:
- 실행한 명령
```

예시:

```text
async 예제 코드에 비동기 처리 기준 주석을 보강한다

- API/View 컨트롤러에 async 예제의 사용 기준과 권장 조합 설명 추가
- AsyncService에 @Async 직접 사용과 CompletableFuture 기반 병렬 처리의 차이 설명 보강
- sptTaskExecutor를 통한 ThreadLocal 컨텍스트 이관과 예외 회수 기준 명시
```
