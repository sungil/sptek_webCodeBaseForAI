# 업무코드영역 코딩스타일 정책

이 문서는 `com.{companyName}.{domain}` 하위의 실제 업무코드영역을 개발할 때의 코딩 스타일을 정의한다.
업무코드영역은 Framework코드영역을 새로 만드는 곳이 아니라, 이미 제공된 Framework코드영역 기능과 예시 프로젝트의 사용 방식을 조합해 실제 업무 기능을 구현하는 영역이다.

새 업무 코드는 먼저 `_webFrameworkExample`의 Controller, Service, DTO, security, view, profile 사용 방식을 확인하고 비슷한 업무 흐름은 같은 형태로 작성한다.
다만 예시 URL, 샘플 권한, 테스트 데이터, 임시 주석은 실제 업무 코드로 그대로 옮기지 않는다.

업무 클래스는 도메인 안에서 역할이 드러나게 배치한다.
Controller는 요청 매핑과 응답 변환, Service는 업무 흐름과 transaction 경계, DTO는 요청/응답 형태, Repository나 Mapper는 데이터 접근 역할에 집중한다.

업무코드영역의 새 패키지 구조를 잡아야 하는 경우에는 `업무코드영역-패키지구성-정책.md`를 먼저 따른다.

업무 코드에서 보안, 공통 응답, 예외 처리, locale, file, message conversion, JSON escape, CORS 같은 기술 처리가 필요하면 먼저 `__webFramework`에 이미 제공된 기능과 annotation 사용 방식을 확인한다.
같은 목적의 filter, interceptor, aspect, converter, helper를 업무 영역에 새로 만들기 전에 기존 기능을 재사용할 수 있는지 판단한다.

변수명은 특별한 이유가 없으면 클래스명이나 타입명을 자연스럽게 낮춘 형태를 사용한다.
업무 의미가 더 중요한 값은 도메인 용어가 드러나게 이름을 짓는다.

```java
private final MemberService memberService;
private final OrderRepository orderRepository;
private final DomainUserDetailsService domainUserDetailsService;
```

환경별로 달라질 수 있는 업무 설정값은 업무 프로젝트의 profile 설정으로 분리한다.
Framework코드영역 공통 동작을 바꾸는 설정인지, 특정 업무 도메인만의 설정인지 먼저 구분한다.

주석은 업무 규칙의 이유, 예외 케이스, 외부 시스템과의 약속처럼 코드만 보고 판단하기 어려운 부분에만 작성한다.
단순히 코드 동작을 다시 설명하는 주석은 작성하지 않는다.

업무 기능을 개발하면서 Framework코드영역 변경이 필요해 보이면 바로 섞어서 수정하지 않는다.
업무 요구를 구현하기 위한 변경인지, 여러 업무 프로젝트에 공통으로 필요한 Framework코드영역 개선인지 먼저 분리해서 판단한다.
