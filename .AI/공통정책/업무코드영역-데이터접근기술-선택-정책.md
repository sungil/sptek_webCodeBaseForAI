# 업무코드영역 데이터접근기술 선택 정책

이 문서는 `com.{companyName}.{domain}` 하위 업무코드영역에서 DB 접근 기능을 만들 때 MyBatis와 JPA를 어떻게 선택하고 조합할지 판단하는 기준을 정의한다.

## 기본 원칙

기존 업무 도메인에 이미 사용 중인 데이터 접근 방식이 있다면 그 방식을 우선한다.
새 기능마다 취향으로 MyBatis와 JPA를 바꾸지 않는다.

기존 기준이 없다면 기능의 성격을 보고 선택한다.
복잡한 SQL 제어가 중요하면 MyBatis를 우선하고, Entity 상태 변경과 도메인 모델 관리가 중요하면 JPA를 우선한다.

## 메인 클래스 설정 확인

DB 접근 기능을 만들기 전에 실행 애플리케이션의 메인 클래스를 확인한다.

- `@Enable_JpaHybrid_At_Main`이 없으면 MyBatis/JDBC transaction manager 기준으로 동작한다.
- `@Enable_JpaHybrid_At_Main`이 있으면 MyBatis와 JPA를 함께 사용할 수 있고, `transactionManager`는 JPA 기준으로 등록된다.
- JPA를 사용하려면 메인 클래스에 `@EnableJpaRepositories`와 `@EntityScan`의 대상 패키지가 업무코드영역에 맞게 선언되어 있어야 한다.
- MyBatis를 사용하려면 profile 설정의 `web-framework.mybatis` mapper XML 위치와 실제 mapper XML 위치가 맞아야 한다.

이 설정이 없거나 범위가 맞지 않으면 코드 작성 전에 어떤 설정을 추가하거나 조정할지 사용자에게 먼저 안내한다.

## MyBatis를 우선하는 경우

다음 기능은 MyBatis를 우선 검토한다.

- 검색 조건이 많은 목록 조회
- 여러 테이블을 조합한 조회
- 통계, 리포트, 대시보드, 엑셀 다운로드
- DB vendor별 SQL 튜닝이나 hint가 필요한 기능
- 기존 SQL이나 mapper XML을 유지해야 하는 기능
- 조회 결과가 Entity보다 화면/API 전용 DTO에 가까운 기능

## JPA를 우선하는 경우

다음 기능은 JPA를 우선 검토한다.

- 단순 등록, 조회, 수정, 삭제
- Entity 상태 변경이 중요한 기능
- transaction 안에서 변경 감지를 활용하는 기능
- 연관관계가 비교적 명확한 업무 모델
- Java 코드에서 도메인 상태와 규칙을 함께 관리하는 편이 자연스러운 기능

## Hybrid 사용 기준

`@Enable_JpaHybrid_At_Main`이 활성화된 애플리케이션에서는 MyBatis와 JPA를 함께 사용할 수 있다.
이 경우 두 기술을 무조건 하나로 합치는 것이 아니라, 역할을 나눠 사용하는 것을 기본으로 한다.

권장 역할 분리는 다음과 같다.

- 상태 변경, Entity 저장, 단순 단건 조회: JPA
- 복잡 조회, 통계, 리포트, SQL 튜닝이 필요한 조회: MyBatis

같은 service transaction 안에서 두 기술을 함께 사용할 수는 있지만, 같은 aggregate나 같은 table을 JPA와 MyBatis가 동시에 변경하지 않는다.
JPA 영속성 컨텍스트의 Entity 상태와 MyBatis 직접 SQL 결과가 어긋날 수 있기 때문이다.

같은 table을 양쪽에서 모두 다뤄야 한다면 한쪽은 조회 전용으로 제한하거나, flush/clear 필요성, transaction 경계, 캐시 영향, 변경 순서를 명확히 검토한다.
이 판단이 필요한 경우 실제 코드 작성 전에 사용자에게 먼저 설명한다.

## 패키지 기준

JPA를 쓰면 `repository` 패키지를 사용한다.
MyBatis를 쓰면 `mapper` 패키지와 mapper XML 위치를 함께 잡는다.

```text
com.{companyName}.{domain}.{businessArea}.repository
com.{companyName}.{domain}.{businessArea}.mapper
```

둘 다 필요한 경우에도 역할이 섞이지 않도록 패키지를 분리한다.

## 사용자 안내 기준

새 업무 기능에서 기존 기준이 없거나 MyBatis/JPA 선택이 애매하면 실제 코드 작성 전에 선택 기준을 사용자에게 먼저 안내한다.
