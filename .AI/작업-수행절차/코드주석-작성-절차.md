## 작성 기준

- 코드만으로 드러나기 어려운 책임 경계, 적용 조건, 예외 조건을 설명한다.
- 오래됐거나 현재 동작과 다른 주석은 현재 코드 기준으로 수정한다.
- 주석으로 문제를 덮지 않는다. 코드 수정이 필요한 문제면 수정 범위와 영향도를 먼저 판단한다.
- 사람이 남긴 주석은 유지하데 주석 내용이 사실과 다른 경우는 내용을 바로 잡을수 있다.


## 클래스 주석

- 클래스에 어떤 성격의 코드가 모이는지, 어디까지가 책임인지 설명한다.
- 메서드 목록이나 변경 이력은 적지 않는다.
- Base 프레임워크 클래스는 여러 업무 프로젝트가 읽는다는 전제로, 사용 조건과 책임 경계를 조금 더 분명하게 쓴다.


## 메서드 주석

- public/protected 메서드, 외부에서 재사용될 package-private 메서드를 우선 검토한다.
- 단순 getter/setter, 단순 위임, 이름만으로 충분한 private helper에는 주석을 추가하지 않는다.
- 예외 조건, 대체 API, thread/request/session/transaction 의존성이 있으면 짧게 남긴다.


## 어노테이션 클레스 주석

- `At_Main`, `At_RestControllerMethod`, `At_Param`, `At_DtoString`처럼 적용 위치가 이름에 들어간 어노테이션은 어디에 붙이는지와 실제 해석 지점을 설명한다.
- `@Target`과 이름의 적용 위치가 맞는지도 함께 확인한다.
